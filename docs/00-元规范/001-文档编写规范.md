# 001 · 文档编写规范

> 本文定义每一篇知识点文档"必须写成什么样"。所有贡献者（包括 AI Agent）在新增或修改文档时必须严格遵循。

## 一、六条硬性要求

本知识库的每篇文档都必须满足以下要求（源自项目核心诉求）：

1. **系统性分类**：文档必须归入某个系统性分类目录（见 [000 总览](./000-分类总览与知识图谱.md)），不允许出现"游离"文档。
2. **章节序号命名**：分类内文档一律以三位序号开头，如 `001-神经网络结构.md`。
3. **知识点全面且正确**：内容必须覆盖该知识点的核心要素，且**保证正确无误**——涉及公式、结论、数据时需可追溯、可验证。
4. **教授级讲解 + 案例**：以严谨专业（大学教授级别）的深度讲解知识点，并配合**真实案例**帮助理解。
5. **通俗化解读**：对相对专业的概念，额外给出**通俗类比或大白话解释**，降低理解门槛。
6. **分类总结与知识图谱**：每个分类必须有一篇 `000` 总结文档，串联本分类知识点并用 Mermaid 绘制**知识图谱**。

## 二、单篇文档推荐结构

为保证系统性与可读性，单篇知识点文档建议采用如下结构（可按需增减小节，但"通俗解读"和"案例"两节强烈建议保留）：

```markdown
# NNN · 知识点标题

> 一句话点明本文要解决的核心问题。

## 一、直觉与通俗解读        # 先用大白话/类比讲清"是什么、为什么"
## 二、严谨定义与原理        # 教授级：定义、公式、推导、前提条件
## 三、案例解析              # 真实/可复现案例，最好配代码或数据
## 四、常见误区与边界        # 易错点、适用范围、反例
## 五、小结与延伸阅读        # 要点回顾 + 关联知识点链接
```

## 三、通俗解读怎么写（要求 5）

- **先直觉后公式**：在给出严谨定义前，先用一句类比说清本质。例如把"梯度下降"比作"蒙眼下山，每一步都朝最陡的下坡方向迈一小步"。
- **少术语、多画面**：能用生活场景说明的，不要堆砌术语。
- **通俗不等于不准确**：类比之后必须回到严谨定义，避免让读者形成错误直觉。

## 四、案例要求（要求 4）

- 案例应**可复现**：给出数据来源、参数、期望结果；涉及代码时放入配套代码目录并在文中引用（见 [002 · 目录与命名规范](./002-目录与命名规范.md)）。
- 优先使用**经典且被广泛验证**的案例（如 MNIST、XOR 问题、房价回归），保证正确性。

## 五、正确性保障（要求 3）

- **公式与结论需可追溯**：重要结论应能对应到教科书/论文/官方文档的共识；有争议处需注明。
- **数值示例需自洽**：文中出现的计算示例应能手算或用配套代码复现。
- **提交前自检**：运行 `npm run docs:validate` 校验命名与结构；本地 `npm run dev` 预览，确认公式、图谱、链接均正常渲染。

## 六、知识图谱写法（要求 6）

- 使用 Mermaid 代码块（```mermaid），本站点已集成渲染。
- 常用图类型：`graph TD`（依赖/层次关系）、`flowchart`（流程）、`mindmap`（发散知识点）。
- 节点命名用**中文知识点名**，箭头表达"依赖 / 演进 / 组成"关系，并用 `classDef` 分层着色以增强可读性。

## 七、小结

- 六条硬性要求是所有文档的底线：分类、序号、全面正确、教授级+案例、通俗解读、分类总结含图谱。
- 单篇文档遵循"直觉 → 原理 → 案例 → 误区 → 小结"的结构。
- 提交前必须通过 `docs:validate` 校验并本地预览。
