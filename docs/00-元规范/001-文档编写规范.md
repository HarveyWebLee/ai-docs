# 001 · 文档编写规范

> 本文定义每一篇知识点文档"必须写成什么样"。所有贡献者（包括 AI Agent）在新增或修改文档时必须严格遵循。
>
> **本知识库面向 AI 开发零基础的初学者**，因此"能不能让小白读懂"是最高优先级。术语堆砌、只讲专业定义、不给大白话，视为**不合格**文档。

## 一、六条硬性要求

本知识库的每篇文档都必须满足以下要求（源自项目核心诉求）：

1. **系统性分类**：文档必须归入某个系统性分类目录（见 [000 总览](./000-分类总览与知识图谱.md)），不允许出现"游离"文档。
2. **章节序号命名**：分类内文档一律以三位序号开头，如 `001-神经网络结构.md`。
3. **知识点全面且正确**：内容必须覆盖该知识点的核心要素，且**保证正确无误**——涉及公式、结论、数据时需可追溯、可验证。
4. **教授级讲解 + 案例**：以严谨专业（大学教授级别）的深度讲解知识点，并配合**真实案例**帮助理解。
5. **通俗优先、专业对齐（本仓库的头等要求）**：面向零基础读者，**每个专业概念/专有名词都必须先用大白话或类比讲清"是什么、为什么、解决什么问题"，再给出严谨定义，最后把专业表述与前面的通俗讲法显式对齐**。详见 [第三节](#三、通俗优先-专业对齐写法（要求-5，最重要）)。
6. **分类总结与知识图谱**：每个分类必须有一篇 `000` 总结文档，串联本分类知识点并用 Mermaid 绘制**知识图谱**。

## 二、单篇文档推荐结构

为兼顾"小白读得懂"与"专业不失真"，单篇知识点文档建议采用**通俗优先**的结构（可按需增减小节，但"大白话开场""打比方""与大白话对齐的专业视角""案例"必须保留）：

```markdown
# NNN · 知识点标题

> 本文用大白话回答：……（点明要解决的核心疑问）

## 一、一句话先说清         # 用一句大白话讲清这个概念"是什么"
## 二、打个比方             # 生活化类比，建立直觉画面
## 三、它到底解决什么问题   # 分「问题1/问题2/…」，多用对比表格与具体例子
## 四、专业视角（与大白话对齐）  # 给出定义/公式，每处都回扣前面的大白话（"也就是前面说的……"）
## 五、案例解析             # 可复现案例，最好配代码或数据
## 六、常见误区与边界       # 易错点、适用范围、反例
## 七、一句话总结           # 用最朴素的话收尾
```

> 结构不必死板照搬，但**顺序原则不可违背：先通俗、后专业、再对齐**。严禁上来就甩公式和术语。

## 三、通俗优先 + 专业对齐写法（要求 5，最重要）

这是本仓库最核心的写作要求。参照以下三步法处理**每一个**专业概念：

### 第 1 步：先用大白话说清

- 用一句"人话"概括本质。例：非线性激活函数——"就是让神经网络能**拐弯**，而不只会**画直线**"。
- 首次出现的专有名词，立刻用括号或小注解释（如"梯度（可以理解为'当前最陡的下坡方向'）"）。

### 第 2 步：打比方 + 讲"解决什么问题"

- 用生活场景类比（流水线、下山、查字典、教小孩……），建立画面感。
- **拆成"它解决了哪几个问题"来讲**，每个问题配一个具体小例子或"朴素做法 vs 更好做法"的对比表格。
- 能画简单的文字流程图（用箭头串起因果）就画，帮助读者串联逻辑。

### 第 3 步：给专业表述，并与大白话对齐

- 再给出严谨定义/公式（保证正确、可追溯）。
- **关键**：给出公式后立刻回扣通俗讲法，如"这个连乘公式，说的就是前面那句'三次线性变换等于一次线性变换'"。让读者知道"专业符号 ↔ 刚才的大白话"是同一件事。

### 写作红线

- ❌ 不要一上来就是定义、公式、术语表。
- ❌ 不要写完专业内容就结束，却不回扣通俗解释。
- ✅ 术语必须"先解释再使用"；能用比喻/例子/表格代替纯文字论述的，优先用。
- ✅ 通俗不等于不准确：类比之后必须回到严谨定义，避免留下错误直觉。

### 参考黄金样板

`docs/03-深度学习基础/001-神经网络结构.md` 中"为什么必须有非线性激活函数"一节是本仓库的**通俗写作黄金样板**：先一句话（能拐弯 vs 画直线）→ 流水线比方 → 分问题（退化成单层 / 分不开 XOR / 拟合非线性）+ 对比表格 → 专业公式并与大白话对齐 → 一句话总结。新写或改写文档时请对照该样板。

## 四、案例要求（要求 4）

- 案例应**可复现**：给出数据来源、参数、期望结果；涉及代码时放入配套代码目录并在文中引用（见 [002 · 目录与命名规范](./002-目录与命名规范.md)）。
- 优先使用**经典且被广泛验证**的案例（如 MNIST、XOR 问题、房价回归），保证正确性。

## 五、正确性保障（要求 3）

- **公式与结论需可追溯**：重要结论应能对应到教科书/论文/官方文档的共识；有争议处需注明。
- **数值示例需自洽**：文中出现的计算示例应能手算或用配套代码复现。
- **提交前自检**：运行 `npm run docs:validate` 校验命名与结构；本地 `npm run dev` 预览，确认公式、图谱、链接均正常渲染。

## 六、知识图谱写法（要求 6）

- 使用 Mermaid 代码块（```mermaid），本站点已集成渲染。
- 常用图类型：`graph TD`（依赖/层次关系）、`flowchart`（流程）、`mindmap`（发散知识点）。
- 节点命名用**中文知识点名**，箭头表达"依赖 / 演进 / 组成"关系，并用 `classDef` 分层着色以增强可读性。

## 七、小结

- 六条硬性要求是所有文档的底线；其中**通俗优先、专业对齐**是本仓库的头等要求。
- 单篇文档遵循"**先通俗（一句话/比方/解决什么问题）→ 后专业（定义/公式）→ 再对齐**"的顺序。
- 每个专有名词都要"先解释再使用"；多用比喻、例子、对比表格；统一译名与释义见 [003 · 专有名词与术语表](./003-专有名词与术语表.md)。
- 提交前必须通过 `docs:validate` 校验并本地预览。
