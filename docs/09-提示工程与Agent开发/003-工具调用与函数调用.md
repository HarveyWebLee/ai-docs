# 003 · 工具调用与函数调用

> 本文用大白话回答：大模型明明很强，为什么还需要"外挂工具"？函数调用到底是模型自己动手，还是别人代劳？MCP 这种协议解决了什么麻烦？让模型能"动真格"之后，怎么保证安全？

## 一、一句话先说清

**工具调用（Tool Calling）= 给只会"动嘴"的模型配上"手和外设"：该查就让它查、该算就让它算、该办事就让它调 API。**

关键分工：**模型只负责"决定用哪个工具、传什么参数"，真正的执行由外部程序来做。**

## 二、打个比方：一个"聪明但被关在屋里"的顾问

大模型很强，但有"**三大天生短板**"，就像一个学识渊博、却被关在没有网络、没有计算器、没有电话的屋子里的顾问：

1. **记忆过时**：训练截止后的新信息它不知道；
2. **不擅精算**：大数运算、精确计算容易出错；
3. **不能行动**：它只会"说"，不能真的发邮件、查数据库、下订单。

**工具调用就是给这位顾问配齐外设**：

| 短板 | 配什么"外设" | 谁来执行 |
| --- | --- | --- |
| 记忆过时 | 搜索工具 | 外部程序去查，再把结果告诉模型 |
| 不擅精算 | 计算器 | 外部程序算好，把精确结果回传 |
| 不能行动 | 各种 API | 外部程序真正发邮件/下单 |

模型负责"决定用什么工具、传什么参数"，工具负责"真正执行并返回结果"。

## 三、它到底解决什么问题

### 问题 1：模型"知道该干嘛"，但没法亲自动手

模型能理解"这道题要算""这事要查"，却不能自己执行。于是有了一套标准配合流程——**函数调用（Function Calling，"模型说要调哪个函数、由程序去执行"）**：

```mermaid
flowchart LR
  U["用户提问"] --> M["模型判断需要调用哪个工具"]
  M --> C["输出结构化调用<br/>函数名 + 参数(JSON)"]
  C --> E["应用真正执行该函数"]
  E --> R["把执行结果回传给模型"]
  R --> A["模型据结果生成最终回答"]
```

**关键点：模型本身不执行函数**，它只输出"要调用哪个函数、参数是什么"（通常是 JSON）；由外部应用真正执行，再把结果喂回模型。

> 打比方：模型像"点菜的客人"，只负责说清要哪道菜、几分熟；"厨房"（外部应用）才真正把菜做出来端回去。

### 问题 2：模型怎么知道有哪些工具、每个工具怎么用

你得先把工具"介绍"给模型，否则它无从选起。这份介绍叫**工具声明（Schema，"工具的说明书"）**，要写清：名称、功能说明、参数结构（JSON Schema）。

> 描述写得越清楚，模型越能正确选择工具、正确填参数。例如一个天气查询工具，需声明参数 `city`（字符串，必填）。

| 朴素做法 | 更好做法 |
| --- | --- |
| 工具描述含糊（"查天气"） | 写明功能 + 参数类型 + 是否必填 |
| 结果：模型选错工具或填错参数 | 结果：模型稳定选对、填对 |

### 问题 3：每家模型各搞一套接口，太乱

如果每个模型、每个应用都自定义一套工具对接方式，工具就没法复用。于是出现了标准协议——**MCP（Model Context Protocol，"模型和工具/数据源之间的通用插座标准"）**：

统一"模型 ↔ 工具/数据源"的对接方式，让工具可复用、可组合（本知识库仓库即接入了 DeepWiki、Context7 等 MCP 服务）。

### 问题 4：模型能"动真格"了，风险也来了

工具调用意味着模型能真正执行操作，安全就成了底线：

- **权限最小化**：只授予必要工具与权限（如只读）。
- **参数校验与确认**：对危险操作（删除、支付）要校验参数并加人工确认。
- **防注入**：警惕**提示注入（Prompt Injection，"外部返回的内容里藏着恶意指令，试图操控模型"）**——外部返回内容里可能藏有恶意指令，试图操控模型。

## 四、专业视角（与大白话对齐）

- **函数调用（Function Calling）**：模型根据用户意图，输出结构化的函数名与参数（通常为 JSON），由外部应用执行后将结果回传，模型再据此生成最终回答。对齐前面：模型只"点菜"，厨房（应用）才"做菜"。
- **工具声明（Schema）**：以 JSON Schema 描述工具的名称、功能与参数结构，供模型据以选择与填参。对齐前面：就是给模型的"工具说明书"。
- **MCP（Model Context Protocol）**：标准化"模型 ↔ 工具/数据源"对接方式的协议，使工具可复用、可组合。对齐前面：就是通用的"插座标准"，免得每家各搞一套。

## 五、案例解析：模型如何"算对"一道大数乘法

**问题**："384762 × 918273 等于多少？"

- **纯靠模型心算**：大概率算错（模型不擅长精确大数运算）。
- **借助工具调用**：
  1. 模型判断这是计算任务，输出调用：`calculator(expression="384762*918273")`；
  2. 应用真正执行该表达式，得到精确结果 `353,316,556,026`；

    （可用 `python3 -c "print(384762*918273)"` 核验，输出 `353316556026`。）
  3. 结果回传，模型据此回答："等于 353,316,556,026。"

**模型负责'理解意图与决定调用'，把'精确执行'交给可靠工具**——这正是工具调用扬长避短的价值：让模型做它擅长的语言理解，把计算交给计算器。

## 六、常见误区与边界

- **误区："模型会自己执行函数"**：模型只**决定**调用并给参数，执行由外部应用完成。
- **误区："给工具越多越好"**：工具过多会让模型选择困难、易误用；应精简并写清描述。
- **误区："工具返回可信"**：外部内容可能含提示注入，需隔离与校验。

## 七、一句话总结

- 工具调用弥补大模型"记忆旧、不精算、不能行动"的短板；模型只决策，应用来执行。
- 清晰的工具 Schema 是正确调用的前提；MCP 等协议实现标准化对接。
- 权限最小化、参数校验、防注入是安全底线。下一步：[004 · Agent 架构与评估](./004-Agent架构与评估.md)。
