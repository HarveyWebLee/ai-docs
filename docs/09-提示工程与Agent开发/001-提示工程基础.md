# 001 · 提示工程基础

> 本文用大白话回答：提示工程到底是啥？为什么"同一个模型、换个问法"结果差那么多？一个好提示由哪些零件拼成？零样本、少样本、System/User/Assistant 这些词到底在说什么？

## 一、一句话先说清

**提示工程（Prompt Engineering，"设计怎么跟大模型说话"的手艺）= 把你的需求说清楚，让模型一次就答到点子上。**

- 你写给模型的那段话叫**提示（Prompt，也就是"你给模型下的指令"）**。
- 提示工程不改模型、不训练模型，只优化"你怎么问"——问法变了，答案质量就可能天差地别。

就这么点事。下面一步步讲透。

## 二、打个比方：给实习生派活

同样一个需求，随便一问和精心设计地问，效果完全不同。就像给一个刚来的实习生派活：

- 含糊地说"帮我写个东西" → 他一头雾水，交上来的多半不是你要的；
- 清楚地说"帮我写一封**面向新客户**的**产品介绍邮件**，语气**专业友好**，**200 字以内**，包含**三个卖点**" → 他一次就能交出满意的稿子。

区别不在实习生的能力，而在**你有没有把话说清楚**。模型也一样：

| 生活里的做法 | 提示工程里的对应 | 效果 |
| --- | --- | --- |
| 交代清楚"做什么" | 写明确的指令 | 模型不跑偏 |
| 给背景资料 | 提供上下文 | 模型有据可依 |
| 说清"要什么格式" | 约束输出格式 | 结果能直接用 |
| 指定"用什么口气、什么身份" | 设定角色/风格 | 语气对味 |

**说清楚：要做什么、背景是什么、给谁看、什么格式、什么约束——这就是提示工程的精髓。**

## 三、它到底解决什么问题

### 问题 1：模型不知道你到底想要什么

模型不会读心，你说得越含糊，它越只能靠"平均猜测"来回答。

| 场景 | 朴素做法（含糊问） | 更好做法（说清楚） |
| --- | --- | --- |
| 让模型翻译 | "翻译一下这段" | "把下面文本翻译成英文，保留专业术语" |
| 让模型总结 | "总结这篇文章" | "用三个要点总结，每点不超过 20 字" |

一个高质量提示通常由这几个"零件"拼成：

| 要素 | 作用 | 例子 |
| --- | --- | --- |
| 指令（Instruction） | 明确任务 | "把下面文本翻译成英文" |
| 上下文（Context） | 提供背景/资料 | 相关文档、对话历史 |
| 输入数据 | 要处理的内容 | 待翻译的文本 |
| 输出格式 | 约束返回形式 | "用 JSON 返回，含 title 和 summary 字段" |
| 角色/风格 | 设定语气 | "你是一位资深律师" |

### 问题 2：任务很特殊时，光"下命令"不够——得"给样板"

有些任务格式很怪，光靠一句指令模型未必领会，这时**给几个例子让它照葫芦画瓢**更管用。这就引出两种问法：

- **零样本（Zero-shot，"零个示例"）**：只给指令、不给示例，直接让模型做。常规任务用它就够。
- **少样本（Few-shot，"少量示例"）**：在提示里塞几个"输入 → 输出"示例，模型照着仿写。

> 打比方：零样本像"直接口头交代任务"；少样本像"再附上两份做好的样板给他参考"。任务越不常规、格式越特殊，给样板（少样本）越有效。

少样本之所以有效，靠的是大模型的**上下文学习（In-Context Learning，"看着提示里的例子临时学会照做"）**能力——**它不需要重新训练、不改动模型参数**，只是从你给的例子里现学现用。

### 问题 3：多轮对话里，"谁在说话"要分清

对话式 API 通常把消息分成三种**角色**，好比一台戏里的不同角色牌：

| 角色 | 大白话 | 作用 | 例子 |
| --- | --- | --- | --- |
| System | "总导演的规矩" | 设定全局身份与规则 | "你是严谨的医学助手，不确定就说不知道" |
| User | "用户这次说的话" | 用户的具体请求 | "帮我看看这个化验单" |
| Assistant | "模型之前的回复" | 多轮时作为对话历史 | 上一轮模型说过的话 |

**合理使用 System 消息，能稳定地约束模型的行为**——相当于开场就把规矩立好，后面每一轮都照这个规矩来。

## 四、专业视角（与大白话对齐）

把前面的大白话收拢成严谨说法：

- **提示工程**：通过设计输入文本（提示），在**不改变模型参数**的前提下，引导大语言模型产生期望输出的方法。对齐前面：就是"只改怎么问，不动模型本身"。
- **零样本 / 少样本**：提示中提供的示例数量为 0 或少量。少样本依赖**上下文学习**——模型在推理时（而非训练时）从提示内的示例归纳出任务模式。对齐前面：就是"照葫芦画瓢，且不用重新训练"。
- **角色消息**：System / User / Assistant 三类消息构成对话上下文，其中 System 提供最高优先级的行为约束。对齐前面：就是"总导演的规矩 + 用户的话 + 模型的历史发言"。

## 五、案例解析：从模糊到精确的提示改造

**任务**：让模型从一段客户评论中提取情感和关键问题。

- **模糊提示**：`分析这条评论：{评论}` → 模型可能返回一大段自由发挥的文字，格式不定、难以程序化使用。
- **精确提示**（角色 + 格式 + 少样本）：

```text
你是评论分析助手。请判断情感（正面/负面/中性）并提取最多 3 个关键问题。
仅返回 JSON，格式：{"sentiment": "...", "issues": ["..."]}

示例：
评论：物流很快但包装破损。
输出：{"sentiment": "负面", "issues": ["包装破损"]}

评论：{待分析评论}
输出：
```

改造后，模型稳定返回结构化 JSON，可直接被下游程序解析。**同一个模型，靠提示设计就把"不可用"变成了"生产可用"**——这正是提示工程的价值。

对照第四节：这里同时用上了**角色**（"你是评论分析助手"）、**输出格式**（JSON 约束）、**少样本**（给了一个示例）三个零件。

## 六、常见误区与边界

- **误区："提示越长越详细越好"**：冗余信息会稀释重点、浪费上下文预算；应精准而非冗长。
- **误区："模型没做好一定是模型笨"**：很多时候是提示没说清；先优化提示再考虑换模型/微调。
- **边界**：提示工程无法凭空注入模型不具备的知识（该用 [RAG](../05-大语言模型与Transformer/004-检索增强生成RAG.md)）或改变其根本能力（该用微调）。

## 七、一句话总结

- 好提示 = 明确指令 + 必要上下文 + 输出格式 + 角色约束；少样本靠示例引导。
- 善用 System 消息稳定行为；先调提示，再考虑 RAG 或微调。
