# 002 · 实验管理与可复现性

> 本文回答：为什么 ML 实验难以复现？要追踪哪些东西才能复刻结果？版本化与实验跟踪工具解决了什么？

## 一、直觉与通俗解读

做机器学习就像做科学实验——**如果一次好结果无法被复现，它就不可信、也无法改进**。

但 ML 实验特别容易"复现失败"，因为影响结果的因素太多：代码、数据、超参数、随机种子、依赖库版本、硬件……改动其中任何一个都可能让结果对不上。

**可复现性**的核心思想很简单：**把每次实验的"全部配方"完整记录下来，任何时候都能一键复刻**。这就是实验管理要解决的问题。

## 二、严谨定义与原理

### 2.1 需要追踪的"四件套"

要复现一次实验，至少需固定并记录：

| 维度 | 内容 |
| --- | --- |
| **代码** | Git commit 哈希（代码版本） |
| **数据** | 数据集版本/快照（如 DVC 管理） |
| **配置** | 超参数、特征列表、随机种子 |
| **环境** | 依赖库版本、镜像（如 Docker）、硬件信息 |

只要这四件套固定，结果就应可复现。

### 2.2 随机性的控制

ML 中随机性来源众多：权重初始化、数据打乱、Dropout、数据增强等。可复现要求**固定随机种子**，并注意某些 GPU 算子存在非确定性，必要时开启确定性模式（可能牺牲速度）。

### 2.3 实验跟踪

用实验跟踪工具（如 MLflow、Weights & Biases）自动记录每次运行的：

- 参数（超参、配置）
- 指标（loss、准确率随步数变化）
- 产物（模型权重、图表）

并支持多次实验横向对比，找出"哪个改动带来了提升"。

### 2.4 模型注册表

训练好的模型登记到**模型注册表**（Model Registry），带版本、阶段（staging/production）、来源实验链接，实现从"实验"到"上线模型"的可追溯治理。

## 三、案例解析：为什么"同样的代码"结果不一样

同事说他的模型准确率 92%，你拉了同一份代码却只跑出 88%，排查发现：

1. 他用的是 `数据集 v2`（清洗过），你用的是 `v1` → **数据版本不同**。
2. 他固定了 `seed=42`，你没设种子 → **随机初始化不同**。
3. 他的 `numpy/torch` 版本与你不同 → **依赖版本不同**。

把三者对齐（同数据版本、同种子、同依赖）后，你也复现出了 92%。**这说明"代码相同"远不足以复现结果**——数据、种子、环境同样是实验的一部分，必须一并记录与固定。这正是实验管理的价值所在。

## 四、常见误区与边界

- **误区："记录代码就够了"**：数据、种子、环境同样关键，缺一不可。
- **误区："复现只为存档"**：可复现是可靠调优、协作与合规审计的基础。
- **边界**：分布式训练、混合精度、GPU 非确定性可能导致微小差异，实践中追求"结果在合理误差内可复现"。

## 五、小结与延伸阅读

- 可复现需固定并记录"代码 + 数据 + 配置 + 环境"四件套，并控制随机种子。
- 实验跟踪工具记录参数/指标/产物并支持对比；模型注册表实现模型治理。
