# 005 · 成本与性能优化

> 本文用大白话回答：模型上线后，老板问"QPS 能到多少、一个月花多少钱"怎么答？量化、批处理、缓存这些优化各自省的是什么？
>
> 读完你会知道：推理性能 = 延迟 × 吞吐 × 资源单价；优化就是**让同样硬件多干活、让每次调用少花钱**。

## 一、一句话先说清

**成本与性能优化 = 在满足业务精度要求的前提下，让模型推理更快、更省算力、更省钱。**

两个核心指标：

- **延迟（Latency）**：一次请求从进来到出结果要多久（毫秒级）。
- **吞吐（Throughput）**：单位时间能处理多少请求（QPS、tokens/s）。

[003 部署与推理服务](./003-模型部署与推理服务.md) 解决"怎么上线"；本文解决"上线后怎么**又快又省**"。

## 二、打个比方：餐厅出餐效率

把推理服务想象成一家餐厅：

| 现象 | 餐厅类比 | 推理系统对应 |
| --- | --- | --- |
| 客人等太久 | 出菜慢 | **延迟高**（P99 超标） |
| 高峰期排长队 | 厨房产能不足 | **吞吐低**、GPU 打满 |
| 每道菜现从头做 | 浪费重复劳动 | 没做**缓存**、没**批处理** |
| 用大锅炒一小份 | 浪费燃气 | 大模型跑小请求、**利用率低** |
| 换更便宜的食材但味道差不多 | 降本 | **量化 / 蒸馏** |

优化目标：**同样口味（精度），更短等待、更高翻台率、更低成本。**

## 三、它到底解决什么问题

### 问题 1：单次推理慢——模型太大、算子太重

| 手段 | 大白话 | 典型收益 |
| --- | --- | --- |
| **模型量化（Quantization）** | 权重从 FP32 压到 INT8/FP16，算法"粗精度"换速度 | 2~4× 加速，显存减半 |
| **知识蒸馏（Distillation）** | 大模型教小模型，小模型模仿大模型输出 | 模型小几倍，精度略降 |
| **剪枝（Pruning）** | 去掉不重要的连接/通道 | 稀疏加速（需硬件支持） |
| **选更小架构** | 换 MobileNet、DistilBERT 等 | 结构性降本 |

量化分 **PTQ（训练后量化）** 与 **QAT（量化感知训练）**；LLM 常用 GPTQ、AWQ 等方案。

> 对齐：量化就是"用稍低精度存权重"，计算更快、占内存更少；蒸馏是"让小弟学大哥"，保留大部分本事但体量更小。

### 问题 2：GPU 利用率低——请求一个一个处理

**朴素做法**：来一条请求跑一条，GPU 大量时间在等 I/O、等 CPU。

**更好做法**：

- **动态批处理（Dynamic Batching）**：短窗口内攒多条请求一起 forward，摊薄启动开销；
- **连续批处理（Continuous Batching，LLM）**：不同长度的生成请求拼批，完成的先走、新的补进；
- **异步流水线**：预处理、推理、后处理并行。

吞吐往往随 batch 增大而提升，但**单请求延迟**也可能增加——需在延迟与吞吐间折中（SLA 驱动）。

### 问题 3：重复计算多——缓存与复用

| 缓存类型 | 适用场景 | 说明 |
| --- | --- | --- |
| **KV Cache（LLM）** | 自回归生成 | 已算过的 key/value 不重复算，生成速度关键 |
| **Embedding 缓存** | 热门 query / 用户 | 相同输入直接返回 |
| **特征缓存** | 推荐、风控 | 离线算好特征，在线只跑轻量模型 |

> 对齐：KV Cache 就像接龙时"前面已经写过的字不用重新想一遍"，是 LLM 推理优化的头号工程。

### 问题 4：怎么算账——成本模型

$$
\text{月成本} \approx \text{实例数} \times \text{实例单价} + \text{存储} + \text{网络 egress}
$$

LLM API 场景常按 **token 数** 计费：

$$
\text{调用成本} = \text{输入 tokens} \times p_{\text{in}} + \text{输出 tokens} \times p_{\text{out}}
$$

优化方向：缩短 prompt、压缩上下文、用小模型路由（简单问题走便宜模型）、缓存高频问答。

**性能监控指标**（与 [004 监控](./004-监控与模型漂移.md) 配合）：

- P50 / P95 / **P99 延迟**
- QPS、tokens/s
- GPU 利用率、显存占用
- 每请求美元成本（$/1K requests）

## 四、专业视角（与大白话对齐）

### 4.1 优化决策矩阵

| 约束 | 优先手段 |
| --- | --- |
| 延迟敏感（实时对话） | 量化 FP16/INT8、KV Cache、小模型、边缘部署 |
| 吞吐敏感（离线批处理） | 大 batch、多实例水平扩展 |
| 成本敏感 | 蒸馏、按需扩缩容、Spot 实例、API 路由 |
| 精度不能降 | QAT 而非粗暴 PTQ、只优化工程不改模型 |

### 4.2 常见推理框架

- **TensorRT / ONNX Runtime**：图优化 + 算子融合 + 量化；
- **vLLM / TGI**：LLM 专用，连续批处理 + PagedAttention；
- **TorchServe / Triton**：通用模型服务，动态批处理。

选型见 [003 部署](./003-模型部署与推理服务.md)；框架会改，但"批处理、量化、缓存"三板斧不变。

## 五、案例解析：量化与批处理的直觉数字

假设某 BERT 分类模型在 GPU 上：

| 配置 | 单条延迟 | 吞吐（batch=1） | 吞吐（batch=32） |
| --- | --- | --- | --- |
| FP32，batch=1 | 8 ms | 125 QPS | — |
| FP16，batch=1 | 5 ms | 200 QPS | — |
| INT8，batch=32 | ~0.38 ms（单条摊销） | — | ~2600 QPS |

说明：

1. **FP16/INT8** 降低单条延迟（5 ms vs 8 ms）；
2. **批处理** 把 32 条合成一次 forward，整批约 12 ms 完成（不是 32×8 ms），单条摊销约 0.38 ms，**总吞吐大幅提升**；
3. 若 SLA 要求 P99 < 10 ms，可能不能无限增大 batch——需压测找甜点。

（数字为示意，实际取决于硬件与模型；上线前务必用真实流量压测。）

成本估算示例：1 张 A10 GPU 约 \$1/小时，稳定跑 200 QPS → 约 17M 次/天 → 单次约 \$0.0000014（约 1.4×10⁻⁶，仅算力，不含人力与存储）。

## 六、常见误区与边界

- **误区："量化一定掉很多精度"**：INT8 PTQ 在多数 CV/NLP 模型上精度损失 <1%；LLM 需更谨慎评估。
- **误区："加 GPU 就线性提速"**：通信、序列化、批调度 overhead 存在；要测**端到端**延迟。
- **误区："只优化模型不管数据"**：超大 prompt、冗余上下文是 LLM 成本第一大户——**输入裁剪**往往比换模型更有效。
- **边界**： regulated 行业（金融、医疗）可能不允许某些精度损失或外部 API，优化手段受合规约束。

## 七、一句话总结

- 性能看延迟与吞吐，成本看算力 × 时间 + token 用量；量化、蒸馏、批处理、KV Cache 是四大杠杆。
- 优化必须带 SLA 与精度回归测试，避免"快了但错了"或"省了但不可用"。
