# 001 · 数据管线与特征工程

> 本文回答：数据管线包含哪些环节？特征工程为什么常比模型更重要？什么是训练-服务偏差与数据泄漏？

## 一、直觉与通俗解读

模型的表现上限，很大程度由**数据**决定，业界有句话："**Garbage in, garbage out（垃圾进，垃圾出）**"。

- **数据管线**就是"**从原始数据到可训练数据的自动化流水线**"：采集 → 清洗 → 转换 → 存储，能稳定、可重复地供给数据。
- **特征工程**是"**把原始数据加工成模型爱吃的形式**"。比如把"出生日期"转成"年龄"、把"时间戳"拆成"星期几/是否节假日"——好特征能让简单模型也表现出色。

## 二、严谨定义与原理

### 2.1 数据管线的典型环节

$$
\text{采集} \to \text{校验/清洗} \to \text{转换/特征化} \to \text{存储(数据集/特征库)} \to \text{供训练与服务}
$$

- **清洗**：处理缺失值、异常值、去重、纠正格式。
- **转换**：归一化/标准化、编码类别变量（one-hot、目标编码）。

### 2.2 常见特征工程手段

| 手段 | 说明 |
| --- | --- |
| 数值缩放 | 标准化/归一化，避免量纲主导（对距离类模型尤为重要） |
| 类别编码 | one-hot、序数编码、目标编码 |
| 特征交叉 | 组合多个特征（如"城市×时段"） |
| 时间特征 | 从时间戳提取星期、节假日、周期性（sin/cos） |
| 缺失处理 | 填充、加"是否缺失"指示位 |

### 2.3 两个关键陷阱

- **数据泄漏（Data Leakage）**：训练时用到了推理阶段拿不到的信息（如把"未来"数据混入训练），会导致离线指标虚高、上线暴跌。典型例子：在划分训练/测试集**之前**就用全量数据算了标准化参数。
- **训练-服务偏差（Training-Serving Skew）**：训练时和线上服务时的特征计算逻辑不一致，导致线上表现与离线不符。**特征库（Feature Store）** 通过统一特征定义来缓解此问题。

## 三、案例解析：一次隐蔽的数据泄漏

某团队预测用户是否会在下月流失，特征里不小心包含了"下月登录天数"。

- **离线评估**：准确率高达 99%，皆大欢喜。
- **上线**：模型几乎无用——因为"下月登录天数"在**做预测的时刻根本还不存在**（那是未来信息）。

这是典型的数据泄漏：**用了预测时不可得的未来特征**。正确做法是严格按时间切分，只使用"预测时刻之前"可获得的信息。这个案例说明：**离线指标好看不等于真的有用，必须警惕泄漏**。

## 四、常见误区与边界

- **误区："有模型就够了，数据随便"**：数据质量与特征往往比模型选择更影响效果。
- **误区："标准化在划分数据集前做"**：会造成信息泄漏；应只用训练集拟合缩放器，再应用到验证/测试集。
- **误区："特征越多越好"**：冗余/噪声特征增加过拟合与维护成本，需做特征选择。

## 五、小结与延伸阅读

- 数据管线保证数据稳定可复现地供给；特征工程把原始数据变成高信息量特征。
- 警惕数据泄漏与训练-服务偏差，特征库有助于统一特征逻辑。
