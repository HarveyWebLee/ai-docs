# 003 · 模型部署与推理服务

> 本文用大白话回答：训练好的模型有哪几种"上岗"方式？在线服务的"快"和"多"怎么取舍？怎么上线才不会一推就全崩？为什么大模型推理要单独优化？
>
> 读完你会知道：训练完只是做出了一个"会预测的文件"，让业务能又快又稳地调用它、还能安全上线，才是部署真正要解决的事。

## 一、一句话先说清

**部署 = 把"会做预测的模型文件"变成业务系统能方便、快速、稳定调用的服务。**

训练好的模型只是一个"**会做预测的文件**"，得让业务系统能调用它才产生价值。常见有两种"上岗"方式，外加一条"稳着上线"的原则：

- **在线服务**：像"**随叫随到的外卖**"，用户发一个请求，立刻返回预测（如推荐、风控），要求低延迟。
- **批处理**：像"**每晚统一备菜**"，定时对大批数据一次性打分（如离线用户画像更新），要求高吞吐、不在乎单条延迟。
- 上线要**稳**：不能新版本一推全崩，因此有灰度、金丝雀等"**先让一小部分流量试用**"的策略。

## 二、打个比方：开一家提供"预测"的店

把模型服务想象成一家店，不同客人有不同的"点单方式"：

| 生活里的场景 | 部署里的对应 | 特点 |
| --- | --- | --- |
| 客人现场点单、马上要吃 | 在线服务 | 讲究"快"（低延迟） |
| 半夜统一把明天的菜备好 | 批处理 | 讲究"量大"（高吞吐），慢点没关系 |
| 新菜先给几桌试吃，好评再全店推 | 金丝雀/灰度上线 | 用"小范围试错"控制风险 |

关键直觉：**"快"和"多"常常要取舍**——想一次处理很多（吞吐高），单个请求可能就得多等一会儿（延迟高）；反之亦然。部署的很多技术，都是在这两者之间找平衡，同时保证"换新菜时不翻车"。

## 三、它到底解决什么问题

### 问题 1：模型可以用在哪些不同场景？——部署形态

不同业务对"快"和"多"的要求不一样，于是有不同的部署形态：

| 形态 | 场景 | 关键指标 |
| --- | --- | --- |
| 在线（实时 API） | 推荐、搜索、风控 | 延迟（latency）、QPS |
| 批处理 | 离线打分、报表 | 吞吐（throughput） |
| 流式 | 实时特征/事件 | 端到端时延 |
| 边缘/端侧 | 手机、IoT | 模型体积、功耗 |

### 问题 2：怎么让别人能调用、还保证环境一致？——服务化与打包

- **变成能调用的接口**：用推理服务框架把模型包成 HTTP/gRPC 接口（如 TorchServe、Triton、FastAPI + 模型）——相当于给后厨开一个"点单窗口"。
- **保证到哪都一个味**：用 **容器（Docker）** 打包模型 + 依赖，保证环境一致，配合 Kubernetes 做弹性伸缩——相当于把整个后厨连锅带料打包成"标准集装箱"，搬到哪都一样，忙时还能多开几家分店。

### 问题 3：怎么让它更快、扛更多请求？——延迟优化技术

- **批处理（batching）**：把并发请求攒成一批一起算，提高 GPU 利用率（牺牲少量延迟换吞吐）——像"凑够一锅一起炒"，更划算。
- **模型压缩**：量化（float32→int8）、剪枝、知识蒸馏，减小体积、加速推理——给模型"瘦身"，跑得更快。
- **缓存**：对重复请求缓存结果——同样的单子来过一次，直接端上之前做好的。

### 问题 4：怎么上线才不会一推就全崩？——安全上线策略

- **金丝雀发布 / 灰度**：先把新模型给少量流量，观察指标无异常再逐步放量。（"先给几桌试吃"。）
- **蓝绿部署**：新旧版本并存，切换出问题可秒级回滚。（"新旧两个后厨都开着，出问题立刻切回旧的"。）
- **A/B 测试**：新旧模型分流对比业务指标，用数据决定是否全量。（"用真实数据比一比，谁好留谁"。）

### 问题 5：为什么大模型推理要单独优化？

LLM 推理是自回归的（逐 token 生成），显存与延迟压力大，常用 **KV Cache**（缓存已算的键值，避免重复计算）、连续批处理、量化等专门优化。用大白话说：大模型是"一个字一个字往外蹦"，每蹦一个字都要回看前文，KV Cache 就是"把前文的计算结果存起来，不用每次重算"。

## 四、专业视角（与大白话对齐）

- **在线服务 / 批处理**：前者追求低**延迟（latency）**，后者追求高**吞吐（throughput）**。
  - 对齐：即"随叫随到的外卖"讲快、"每晚统一备菜"讲量。
- **批处理（batching）优化**：把并发请求合批计算以提高硬件利用率，用少量延迟换更高吞吐。
  - 对齐：这就是"凑够一锅一起炒"，也解释了为何"快"与"多"要取舍。
- **金丝雀发布**：先以小流量验证新版本，指标正常再放量。
  - 对齐：即"新菜先给几桌试吃"，把风险关在小范围里。

## 五、案例解析：金丝雀发布如何"兜住"一次事故

新版风控模型离线指标更好，团队用金丝雀发布上线：

1. 先切 **5% 流量**到新模型，95% 仍走旧模型。
2. 监控发现新模型把大量正常交易误判为欺诈（误报率飙升）——这是离线评估没暴露的问题。
3. 由于只影响 5% 流量，**损失被控制在很小范围**，团队立即回滚，用户几乎无感。

如果直接 100% 全量上线，这次误判就会波及所有用户，造成严重事故。**金丝雀发布用"小流量试错"把上线风险降到最低**，这正是安全部署策略的价值。

## 六、常见误区与边界

- **误区："离线好就能直接全量上线"**：线上分布、时延、并发与离线不同，需灰度验证。
- **误区："量化必然掉精度不能用"**：合理的 int8 量化通常精度损失很小，收益显著。
- **误区："部署完就结束了"**：上线只是开始，还需持续监控（见 [004](./004-监控与模型漂移.md)）。

## 七、一句话总结

- 部署形态分在线/批/流/端侧，各有延迟与吞吐取向；容器化保证环境一致。
- 用批处理、量化、缓存优化延迟；金丝雀/蓝绿/A-B 保障安全上线；大模型靠 KV Cache 等专门优化。
