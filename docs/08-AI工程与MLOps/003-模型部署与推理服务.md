# 003 · 模型部署与推理服务

> 本文回答：模型有哪些部署形态？在线服务的延迟/吞吐如何权衡？有哪些安全上线策略？大模型推理为何要特殊优化？

## 一、直觉与通俗解读

训练好的模型只是一个"**会做预测的文件**"，要产生价值，得让业务系统能**方便、快速、稳定地调用它**——这就是部署。

- **在线服务**：像"**随叫随到的外卖**"，用户发一个请求，立刻返回预测（如推荐、风控），要求低延迟。
- **批处理**：像"**每晚统一备菜**"，定时对大批数据一次性打分（如离线用户画像更新），要求高吞吐、不在乎单条延迟。
- 上线要**稳**：不能新版本一推全崩，因此有灰度、金丝雀等"**先让一小部分流量试用**"的策略。

## 二、严谨定义与原理

### 2.1 部署形态

| 形态 | 场景 | 关键指标 |
| --- | --- | --- |
| 在线（实时 API） | 推荐、搜索、风控 | 延迟（latency）、QPS |
| 批处理 | 离线打分、报表 | 吞吐（throughput） |
| 流式 | 实时特征/事件 | 端到端时延 |
| 边缘/端侧 | 手机、IoT | 模型体积、功耗 |

### 2.2 服务化与打包

- 用推理服务框架把模型包成 HTTP/gRPC 接口（如 TorchServe、Triton、FastAPI + 模型）。
- 用 **容器（Docker）** 打包模型 + 依赖，保证环境一致，配合 Kubernetes 做弹性伸缩。

### 2.3 延迟优化技术

- **批处理（batching）**：把并发请求攒成一批一起算，提高 GPU 利用率（牺牲少量延迟换吞吐）。
- **模型压缩**：量化（float32→int8）、剪枝、知识蒸馏，减小体积、加速推理。
- **缓存**：对重复请求缓存结果。

### 2.4 安全上线策略

- **金丝雀发布 / 灰度**：先把新模型给少量流量，观察指标无异常再逐步放量。
- **蓝绿部署**：新旧版本并存，切换出问题可秒级回滚。
- **A/B 测试**：新旧模型分流对比业务指标，用数据决定是否全量。

### 2.5 大模型推理的特殊性

LLM 推理是自回归的（逐 token 生成），显存与延迟压力大，常用 **KV Cache**（缓存已算的键值，避免重复计算）、连续批处理、量化等专门优化。

## 三、案例解析：金丝雀发布如何"兜住"一次事故

新版风控模型离线指标更好，团队用金丝雀发布上线：

1. 先切 **5% 流量**到新模型，95% 仍走旧模型。
2. 监控发现新模型把大量正常交易误判为欺诈（误报率飙升）——这是离线评估没暴露的问题。
3. 由于只影响 5% 流量，**损失被控制在很小范围**，团队立即回滚，用户几乎无感。

如果直接 100% 全量上线，这次误判就会波及所有用户，造成严重事故。**金丝雀发布用"小流量试错"把上线风险降到最低**，这正是安全部署策略的价值。

## 四、常见误区与边界

- **误区："离线好就能直接全量上线"**：线上分布、时延、并发与离线不同，需灰度验证。
- **误区："量化必然掉精度不能用"**：合理的 int8 量化通常精度损失很小，收益显著。
- **误区："部署完就结束了"**：上线只是开始，还需持续监控（见 [004](./004-监控与模型漂移.md)）。

## 五、小结与延伸阅读

- 部署形态分在线/批/流/端侧，各有延迟与吞吐取向；容器化保证环境一致。
- 用批处理、量化、缓存优化延迟；金丝雀/蓝绿/A-B 保障安全上线；大模型靠 KV Cache 等专门优化。
